WIN-CONFIDENCE SCORING RUBRIC — Xero App & Agent Hackathon (0-100 per factor)

**F1 · Xero-Centrality (25%)** — maps to the 50% judging pillar.
- 0: Xero is a peripheral data export; idea works fine with QuickBooks or a spreadsheet. Ledger is optional.
- 50: Xero is the system of record read from, but the core value could live elsewhere; Xero writes are cosmetic.
- 100: Remove Xero and the product ceases to exist. Xero's ledger is the source of truth AND the destination of every action (reads drive decisions, writes execute them).

**F2 · API-Depth (15%)** — the 30% API pillar.
- 0: One list-* read call, no writes.
- 50: Several reads + one write type (e.g. create-invoice), single-object, no idempotency.
- 100: Coordinated multi-endpoint reads feeding writes across ≥3 create/update types (e.g. manual-journal + payment + credit-note), correct accounting pattern (clearing-account gross-up), idempotent, human-in-loop before ledger writes.

**F3 · Weekend-Feasibility (20%)**
- 0: Needs a live external connector/OAuth to a third party (Treatwell, bank) or a charging rail that doesn't exist in scope.
- 50: Buildable in 24h but golden path is fragile or depends on ≥2 shaky moving parts.
- 100: One flow, fully demoable on the resettable UK Demo Company via Custom Connection M2M; every dependency is mocked-input-safe (CSV/PDF upload, not live scrape); stays under 60/min rate limit.

**F4 · Demo-Wow / Legibility (15%)** — the tie-breaker.
- 0: Requires >3 min of setup narration before anything visible happens; outcome is abstract.
- 50: Works but the "aha" needs explanation; judge must trust a number they can't see change.
- 100: A 90-second single flow where a messy input becomes a correct Xero ledger state ON SCREEN — before/after ledger diff is the wow, self-evident to an accountant judge.

**F5 · Differentiation-vs-incumbents (15%)**
- 0: Rebuilds a saturated tool (Hubdoc/Dext OCR, Chaser reminders, Float forecast) or something JAX already does.
- 50: Adjacent to an incumbent but with one genuine twist; overlap is arguable.
- 100: Lands in verified white-space (service/gig payout recon, forecast-and-act CFO copilot, AP statement recon, cleanup-with-audit-trail) — zero incumbent AND falls inside a JAX contractual ban (forecasting/advice) or autonomously FIXES where others only flag.

**F6 · Real-World-Impact / Scale (10%)**
- 0: Niche edge case, few businesses, low money at stake.
- 50: Real pain, mid-size segment, no hard number attached.
- 100: Quantified pain across a large Xero-reachable segment (e.g. late payments £11bn/yr, payments fraud 76% of orgs), plausible App Store distribution to millions of SMEs.

**Anti-gaming notes**
1. *F1 inflation:* teams claim "Xero-central" while only reading. Catch: apply the deletion test — if swapping Xero for a spreadsheet leaves the demo intact, cap F1 at 40 regardless of self-description.
2. *F2 inflation:* padding endpoint count with trivial redundant reads. Catch: score only DISTINCT write types that alter ledger state and are exercised in the live demo; unused SDK calls score zero.
3. *F5 inflation:* relabeling an incumbent's feature as novel. Catch: name the specific incumbent/JAX capability it overlaps; if none can be excluded by a verified fact, treat overlap as present and cap F5 at 50.

## 3. Independent verifier — grounding verdict

**Overall:** This deliverable is unusually disciplined for a hackathon idea-ranking: it self-flags its own unverifiable statistics, shows red-team corrections, and hedges where sources are thin — which reduces (but does not eliminate) hallucination risk. The Xero Accounting REST API has supported POST/PUT to create and update repeating-invoice templates since ~2022; the GitHub issue #113 it cites is scoped to the xero-mcp-server wrapper lacking that tool, NOT to the core API being read-only. ## 4. Full idea pool (all 32 distinct candidates generated, with sources)

- **Gig & Service-Platform Payout Untangler** _(B01)_ — LLM reads any raw payout CSV/PDF from Treatwell, Fresha, Booksy, Uber, Deliveroo, Airbnb, Fiverr or Upwork and gross-ups it into Xero via a clearing account — no bank feed misbooking net-as-revenue.
  - problem: Service/gig platforms pay out NET (after commission/fees/VAT/refunds). If the owner just reconciles the bank deposit as revenue, gross sales, commission expense and VAT are all wrong. There is no pre-built connector for this vertical.
  - xero angle: Agent parses the payout doc, creates a manual-journal (or bank-transaction) that debits a 'Platform Clearing' tracking-category account for gross, credits commission/fee expense + VAT, nets to the actual bank deposit — using create-manual-journal / create-bank-transaction + tracking-category MCP tools, human-approves before posting.
  - why open: Perplexity confirmed (2026): A2X and Link My Books remain ecommerce-only (Amazon/Shopify/eBay/Etsy/Walmart); no documented Xero App Store integration covers Treatwell/Fresha/Booksy/Uber/Deliveroo/Airbnb/Fiverr/Upwork with gross-up clearing-account accounting.
  - sources: perplexity_ask 2026 query on A2X/Link My Books scope (citations: linkmybooks.com, qoblex.com, synder.com, apideck.com)
- **Rolling Forecast Copilot (JAX can't forecast)** _(B03)_ — Reads live Xero P&L, AR aging and each customer's real payment-lateness history to build a rolling 13-week cash-flow forecast with scenario toggles, then autonomously acts on it (drafts chase emails, flags invoices to delay paying).
  - problem: Owners have no forward view of cash — Float/Fathom give read-only static forecasts, and Xero's own JAX agent is explicitly kept out of forecasting/advice per Xero's public positioning.
  - xero angle: Pulls list-invoices, list-aged-receivables/payables, list-bank-transactions via Xero MCP; forecast model weights each customer's historic days-to-pay; on low-cash-alert, agent auto-drafts (not sends) a create-invoice reminder or proposes delaying a bill — human approves the write action.
  - why open: Perplexity confirmed: Xero's own AI-in-accounting guides describe JAX around 'analytics and trend identification' and insights, not forward-looking forecasting — forecasting is not part of JAX's advertised scope, leaving room for a 3rd-party forecasting+action agent.
  - sources: perplexity_ask 2026 query on JAX scope (xero.com/au and /us accountant-bookkeeper-guides/ai-in-accounting)
- **AP Supplier-Statement Reconciler** _(B01)_ — Upload a supplier's monthly statement PDF; agent matches every line against Xero bills, flags missing/duplicate/mismatched bills, and drafts the correcting bill or a chase-supplier email.
  - problem: No native Xero feature checks a supplier statement against what's actually been entered as bills — missing bills silently understate liabilities and duplicate entries overstate spend (duplicate invoices average 1.29% of AP, $2,034 each).
  - xero angle: Parses PDF/CSV statement -> list-bills/list-bank-transactions via MCP to cross-check -> create-bill (draft, unapproved) for anything missing, human approves before posting; discrepancy report is the demo centerpiece.
  - why open: Confirmed by the incumbent list: Booke AI/XBert only flag issues, they do not autonomously fix with an audit trail — no product does statement-vs-Xero-bills reconciliation with a corrective write action.
  - sources: hackathon grounding facts (incumbent list); duplicate-payment stat pre-verified in grounding context
- **Books-Cleanup Remediator with Audit Trail** _(B01)_ — Scans the whole ledger for common month-end errors (miscategorized transactions, unreconciled items, missing tracking categories) and FIXES them via correcting journal entries — with a full before/after audit trail, not just a flag.
  - problem: XBert/Booke AI only surface a list of suspected errors; a human still has to go in and manually fix each one, which small-business owners without a bookkeeper don't know how to do.
  - xero angle: list-bank-transactions/list-trial-balance/list-accounts via MCP to detect anomalies (rule + LLM-classified); proposes create-manual-journal corrections in a batch, owner approves each, agent posts and logs a reversible audit trail (idempotency key per fix).
